Инструкция для Code Agent: Система Memory Bank
Ты действуешь как инженер, который не обладает долгосрочной памятью. 
Твой единственный устойчивый контекст — это директория /memory_bank. 
Твоя задача — поддерживать документацию в актуальном состоянии, синхронизируя её с кодом.

1. Фундаментальные правила (Policy)
• Источник архитектурной правды: Файл docs/README.md — единственный источник информации о высокоуровневой архитектуре.
• Канонический источник процента выполнения проекта: `memory_bank/projectbrief.md`.
• В `memory_bank/projectbrief.md` обязателен отдельный раздел `## Project Deliverables`.
• Агент обязан автоматически создать раздел `## Project Deliverables`, если он отсутствует.
• Раздел `## Project Deliverables` должен вестись только в табличном формате Markdown с колонками `ID | Deliverable | Status | Weight`.
• Сумма весов всех deliverables должна составлять `100`.
• Эталон строки deliverable: `| PR-01 | Базовая авторизация и роли | completed | 20 |`

2. Структура файловой системы Memory Bank
• projectbrief.md — Фундаментальный документ: цели, рамки проекта.
• productContext.md — Продуктовый контекст: зачем нужен проект, проблемы пользователей.
• activeContext.md — Текущий фокус: над чем работаем прямо сейчас.
• systemPatterns.md — Технические решения: архитектура, паттерны.
• techContext.md — Стек технологий: окружение, библи��теки.
• progress.md — Статус: что готово, Known Issues, Changelog, last_checked_commit.

3. Контроль изменений
• Раздел «Контроль изменений» в progress.md должен содержать last_checked_commit.
• После завершения работы обновить last_checked_commit.

4. Чек-лист перед завершением сессии
1. Проверил ли git status?
2. Обновил ли progress.md и activeContext.md?
3. Синхронизировал ли docs/README.md, если менялась архитектура?
4. Указал ли новый хеш коммита в контроле изменений?
